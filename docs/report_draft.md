# Metoy 科学小导师：面向科学教具学习的 LLM Agent

## 项目亮点

本项目构建了一个面向学生科学学习场景的轻量级 EduToy Agent。系统结合 GLM 大模型、本地 RAG 知识库、教具目录工具、上下文记忆和开发者可解释界面，让学生可以用自然语言提问科学知识，并在需要时获得基于真实教具说明书的实验指导。相比普通聊天机器人，本项目重点解决了三个问题：教具不能编造、学生追问要能接上下文、Agent 的内部流程要能被展示和解释。

## 一开始想做什么

最初我在几个方向之间选择：自动生成学习卡片、个性化写作助手、高考志愿辅助工具。后来发现高考志愿需要大量实时招生数据和政策核验，数据维护成本较高；写作助手虽然真实需求强，但展示 Agent 与 RAG 结构不够直观。最终我选择做一个“科学教具学习 Agent”：学生直接聊天，Agent 根据问题判断是讲知识点、查教具，还是设计实验；开发者可以看到检索依据和执行轨迹。

这个方向适合本次大作业，因为它能清楚展示 LLM 应用里的几个核心技术点：系统提示词、RAG、本地工具、Agent 流程、上下文记忆和质量校验。

## 系统做出了什么

当前系统是一个可运行的 Streamlit Web 应用，包含三个主要入口：

- 学生聊天界面：学生像使用 ChatGPT 一样提问。
- 开发者控制台：展示 Agent 架构、运行轨迹、RAG 命中文档和质量校验。
- 教具商店页面：作为产品化场景展示，说明 Agent 可以和教具内容、课程服务结合。

学生可以提出类似问题：

- 彩虹是怎么形成的？
- 你们有哪些教具？
- 用平面镜成像演示仪带我做实验。
- 怎么验证铁和棉花一样重？
- 下一步怎么做？

系统会根据问题类型选择不同路径。如果只是问知识点，它会先解释概念，不会立刻设计实验；如果问教具清单，它会从本地教具说明书目录中读取；如果要求实验，它会结合 RAG 资料生成实验步骤、安全提醒和下一步挑战。

## 数据和知识库

系统使用了本地资料构建轻量知识库，资料来源包括：

- 小学科学教学用书
- 初高中物理教材
- 中学物理实验教学论文
- 益智玩具与教学应用论文
- 教具说明书，例如平面镜成像演示仪、杠杆、液体压强、光的偏振、光的干涉等

原始 PDF 和 Word 文件较大，不直接用于线上部署。系统通过脚本将资料处理成 `data/edutoy/documents.jsonl`，当前知识库包含 192 条文档片段。线上部署时只需要上传这个轻量 JSONL 文件，Agent 就可以进行本地检索。

## Agent 架构

系统的 Agent 流程如下：

```text
User Message
  -> Profile / State
  -> Memory Resolver
  -> Router
  -> Tool Registry
  -> Local RAG / Teaching Aid Catalog
  -> Reason / Experiment Design
  -> Safety Check
  -> GLM or Rule Generator
  -> Quality Verifier
  -> Student UI / Developer Console
```

各模块作用如下：

- Profile / State：整理学生问题、对话历史、学段和约束。
- Memory Resolver：处理“下一步”“这个”“怎么验证呢”等追问。
- Router：判断用户是在问知识点、查教具、做实验，还是问题太宽泛。
- Tool Registry：根据意图决定可用工具，例如本地 RAG、教具目录、质量校验。
- Local RAG：从教材、论文和说明书中检索相关资料。
- Teaching Aid Catalog：回答教具清单时，只读取本地说明书，避免模型编造。
- Reason / Experiment Design：把检索材料转成适合学生的解释或实验设计。
- Safety Check：给出材料和实验安全边界。
- Generator：调用 GLM 或本地规则生成最终回答。
- Quality Verifier：检查回答是否符合“资料有依据、教具不编造、知识点不误入实验”等约束。

## 模型和工具

系统主要使用：

- Python + Streamlit 构建 Web 应用。
- 智谱 GLM API 作为大模型生成模块。
- 本地 JSONL 知识库实现轻量 RAG。
- 自定义 Agent 流程实现意图识别、工具调用、质量校验和开发者可视化。

如果没有配置 GLM API Key，系统会使用本地规则兜底；配置后会进入更自然的智能生成模式。

## 实现中的关键问题

### 1. 不能把学生所有问题都变成实验

一开始系统遇到一个问题：学生只是问“彩虹怎么形成”，Agent 却直接开始设计实验。这不符合真实学习场景。后来我加入了 Router：如果用户只是问知识点、定义或原理，系统先讲概念，最后再询问是否要用教具或实验验证。

### 2. 教具必须准确，不能让模型自由编

当学生问“你们有哪些教具”时，这不是开放生成题，而是标准目录查询。因此系统单独设计了 Teaching Aid Catalog 工具，只从本地教具说明书中列出已有教具，并在回答中说明“来自本地说明书，不是模型猜的”。

### 3. 上下文追问容易串话

在测试中，用户问完“铁和棉花一样重怎么验证”，再问“怎么验证呢”，系统曾错误跳回更早的“麦克斯韦方程组”话题。这个问题说明简单关键词规则不足以处理对话记忆。因此我增加了 Memory Resolver 和最近上下文保护机制，让短追问优先接最近一轮助手回答。

### 4. 用户体验影响智能体感受

如果答案一次性突然出现，用户会觉得慢和突兀。因此界面增加了类似 GPT 的聊天布局、历史对话、删除历史、紧凑思考状态和逐字输出效果。虽然底层还不是真正 token streaming，但体验上更接近真实聊天 Agent。

## 有意思的观察

这个项目让我发现，做一个能用的 Agent 并不是只接一个大模型 API。真正影响效果的部分包括：

- 数据怎么组织；
- 哪些问题应该检索；
- 哪些问题应该调用工具；
- 哪些地方不能让模型自由发挥；
- 对话历史如何压缩和引用；
- 用户界面如何让人相信系统正在认真处理问题。

尤其是教育场景中，模型的“会说”不是全部。它还必须知道自己有哪些教具、资料来自哪里、什么时候该讲知识点、什么时候才进入实验设计。

## 当前不足

当前系统仍然是 MVP，主要不足包括：

- RAG 还是轻量 JSONL 检索，没有接入向量数据库和重排模型。
- 上下文记忆主要针对短会话，还没有长期学生画像。
- 工具调用是自定义流程，没有使用 LangGraph 等正式图编排框架。
- 还没有系统评测集，只进行了人工测试。
- 流式输出是前端模拟，不是真正 token streaming。
- 部署版本还没有用户登录、权限管理和后台日志。

## 如果再给一周

如果继续迭代，我最想做四件事：

1. 引入向量数据库和重排，提高 RAG 命中质量。
2. 做 20 到 30 个固定测试问题，评估知识讲解、教具查询、实验设计和上下文追问。
3. 实现真正的流式输出，让 GLM 的 token 边生成边显示。
4. 增加学生画像和教师后台，让系统更适合真实学校试用。

## 总结

本项目从一个开放的大作业题目出发，最终完成了一个可以运行和展示的 EduToy Agent 原型。它不是单纯的聊天框，而是包含系统提示词、本地知识库、RAG 检索、工具约束、上下文记忆、质量校验和开发者可解释界面的完整 LLM 应用。虽然距离商业级教育产品还有距离，但作为大作业，它已经能展示一个从想法到系统实现再到反思迭代的完整探索过程。
